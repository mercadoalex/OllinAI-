# Requirements Document

## Introduction

OllinAI is a Change Intelligence and Deployment Risk platform designed for engineering teams that want to reduce change failure rates, track DORA metrics, and understand the risk profile of every deployment. The platform ingests deployment events from CI/CD pipelines, correlates them with production incidents, computes risk scores, and provides actionable recommendations to improve software delivery performance.

Target users are engineering managers, platform team leads, and VPs of Engineering at mid-market companies (50–500 engineers) who need data-driven insights into their deployment practices.

## Glossary

- **OllinAI**: The Change Intelligence and Deployment Risk SaaS platform (from Nahuatl "Ollin" meaning change/movement + AI for artificial intelligence)
- **Deployment_Event**: A record of a code deployment containing commit SHAs, authors, services affected, timestamp, and environment
- **Incident**: A production issue reported via an integrated alerting system such as PagerDuty or OpsGenie
- **Risk_Score**: A computed assessment (low, medium, high, critical) of a deployment's likelihood to cause a production incident
- **DORA_Metrics**: The four key metrics from the DORA (DevOps Research and Assessment) framework: Deployment Frequency, Lead Time for Changes, Change Failure Rate, and Mean Time to Recovery
- **Change_Failure_Rate**: The percentage of deployments that result in a correlated production incident
- **Lead_Time**: The elapsed time from first commit to production deployment
- **Deployment_Frequency**: The number of deployments per unit time per team or service
- **MTTR**: Mean Time to Recovery — the average duration from incident detection to resolution
- **Correlation_Window**: The configurable time window after a deployment during which incidents are considered potentially caused by that deployment
- **Service**: A distinct deployable unit within a customer's software architecture
- **Team**: A group of engineers responsible for one or more Services
- **Webhook**: An HTTP callback used to receive Deployment_Events and Incidents from external systems
- **Risk_Factor**: An individual signal contributing to the overall Risk_Score (e.g., change size, deployment time, historical failure rate)
- **Recommendation**: An actionable suggestion generated by OllinAI to reduce deployment risk
- **Tenant**: A customer organization using OllinAI under a subscription plan
- **Retention_Period**: The duration for which historical data is stored, determined by subscription tier
- **OllinAI_Agent**: A lightweight eBPF-based agent installed on CI/CD runners or production hosts that collects deep runtime telemetry
- **Collector_API**: The OllinAI endpoint that receives telemetry batches from OllinAI_Agent instances
- **Syscall_Profile**: A statistical fingerprint of a Service's system call patterns used as a behavioral baseline
- **Observation_Window**: The configurable period after deployment during which the OllinAI_Agent monitors for anomalies
- **AIOps_Engine**: The OllinAI subsystem responsible for training, evaluating, and serving ML models that provide predictive intelligence and automated decision-making for IT operations
- **ML_Model**: A machine learning model trained on historical Deployment_Events, Incidents, and operational telemetry to predict outcomes and detect anomalies
- **Prediction_Score**: A probabilistic assessment (0.0 to 1.0) representing the ML_Model's confidence that a specific outcome (incident, anomaly, failure) will occur
- **Training_Pipeline**: The automated workflow that continuously retrains ML_Models on new data, validates model performance against a holdout set, and promotes models to serving when quality thresholds are met
- **Anomaly_Signal**: A deviation from learned behavioral patterns detected by an ML_Model, including metric anomalies, log pattern anomalies, and deployment pattern anomalies
- **Feature_Vector**: The structured numerical representation of a Deployment_Event, Service, or Incident used as input to an ML_Model for inference
- **Model_Registry**: The versioned store of trained ML_Models including metadata such as training date, dataset version, performance metrics, and deployment status (staging or production)
- **Drift_Score**: A measurement (0.0 to 1.0) of how much current input data distributions have diverged from the data used to train the active ML_Model
- **Remediation_Action**: An automated or recommended operational action generated by the AIOps_Engine (e.g., rollback, scale-up, canary rejection, deployment gate hold)
- **Process_Ancestry**: The parent-child process tree showing the execution lineage of a process within a CI/CD pipeline job (e.g., shell → npm install → node → credential-read)
- **Build_Attestation**: A cryptographically signed document generated per pipeline run that records all processes executed, network connections made, and files written during the run, conforming to the in-toto attestation framework
- **Rule_Bundle**: A versioned, OCI-packaged collection of detection rules that can be distributed and updated independently of the OllinAI_Agent binary
- **Data_Residency_Mode**: A configuration option allowing Enterprise tenants to route OllinAI_Agent telemetry directly to their own Amazon S3 bucket with OllinAI reading from there via cross-account IAM roles, ensuring sensitive data never transits OllinAI infrastructure
- **Kernel_Compatibility_Check**: A pre-flight validation performed by the OllinAI_Agent at startup that verifies the host kernel version, BPF syscall availability, required BPF helper support, and process capabilities before attempting to attach eBPF probes
- **OWASP_CICD_Top_10**: The OWASP Top 10 CI/CD Security Risks framework, a standardized classification of the most critical security risks in CI/CD pipelines (CICD-SEC-1 through CICD-SEC-10), used by OllinAI to contextualize detections within an industry-recognized risk taxonomy

## Requirements

### Requirement 1: Deployment Event Ingestion

**User Story:** As a platform engineer, I want OllinAI to ingest deployment events from my CI/CD pipelines, so that I have a complete record of all deployments across my organization.

#### Acceptance Criteria

1. WHEN a Webhook payload is received at the deployment ingestion endpoint, THE OllinAI SHALL validate the payload schema and persist the Deployment_Event within 2 seconds, returning an HTTP 201 response with the assigned event identifier
2. THE OllinAI SHALL accept Deployment_Events containing the following required fields: at least one commit SHA (maximum 50 per event), author identifier, at least one affected Service name (maximum 20 per event), deployment timestamp, and target environment; and the following optional fields: lines added, lines removed, and files changed count as change size metadata
3. IF a Webhook payload fails schema validation or omits a required field, THEN THE OllinAI SHALL return an HTTP 400 response with an error message identifying each invalid or missing field by name
4. THE OllinAI SHALL support concurrent ingestion of Deployment_Events from at least 20 simultaneous CI/CD sources for a single Tenant without data loss
5. IF a duplicate Deployment_Event is received (matching commit SHA, Service, and environment), THEN THE OllinAI SHALL deduplicate, retain only the first occurrence, and return an HTTP 200 response indicating the event was already recorded
6. WHEN a Deployment_Event is persisted, THE OllinAI SHALL associate the event with the originating Team based on Service ownership configuration
7. IF a Deployment_Event references a Service with no owning Team, THEN THE OllinAI SHALL persist the event with an unassigned Team association and associate it with the correct Team once ownership is configured

### Requirement 2: Incident Ingestion and Correlation

**User Story:** As an SRE, I want production incidents automatically correlated with recent deployments, so that I can identify which changes caused production issues.

#### Acceptance Criteria

1. WHEN an Incident payload is received via Webhook, THE OllinAI SHALL validate and persist the Incident with severity (low, medium, high, critical), affected Service, detection timestamp, and optional resolution timestamp within 2 seconds
2. WHEN an Incident is persisted, THE OllinAI SHALL query Deployment_Events within the configured Correlation_Window preceding the Incident detection timestamp for the affected Service and create correlation links to each matching Deployment_Event
3. IF no custom Correlation_Window is configured by the Tenant, THEN THE OllinAI SHALL use a default Correlation_Window of 60 minutes
4. WHERE a Tenant configures a custom Correlation_Window, THE OllinAI SHALL accept values between 5 minutes and 24 hours and apply that window for all future incident correlation calculations
5. WHEN multiple Deployment_Events fall within the Correlation_Window for a single Incident, THE OllinAI SHALL rank correlations by temporal proximity (most recent deployment first)
6. IF an Incident payload fails validation, THEN THE OllinAI SHALL return an HTTP 400 response with an error message identifying the invalid fields
7. WHEN no Deployment_Events are found within the Correlation_Window for a persisted Incident, THE OllinAI SHALL persist the Incident with zero correlation links and mark it as uncorrelated
8. WHEN an Incident is persisted without a resolution timestamp, THE OllinAI SHALL accept the Incident and update the resolution timestamp when a subsequent Webhook payload provides it for the same Incident

### Requirement 3: DORA Metrics Computation

**User Story:** As an engineering manager, I want to view DORA metrics for my teams and services, so that I can measure and improve software delivery performance.

#### Acceptance Criteria

1. THE OllinAI SHALL compute Deployment_Frequency as the count of Deployment_Events per Service per Team within a specified time range
2. THE OllinAI SHALL compute Lead_Time as the elapsed time in hours from the earliest commit SHA in a Deployment_Event to the deployment timestamp
3. THE OllinAI SHALL compute Change_Failure_Rate as the percentage of Deployment_Events with correlated Incidents to total Deployment_Events within a specified time range
4. THE OllinAI SHALL compute MTTR as the average elapsed time in hours from Incident detection timestamp to resolution timestamp within a specified time range, excluding Incidents that have no resolution timestamp from the calculation
5. WHEN a user requests DORA_Metrics, THE OllinAI SHALL return results filterable by Team, Service, environment, and time range, applying a default time range of 30 days when no range is specified and enforcing a maximum queryable range of 365 days
6. WHEN new Deployment_Events or Incidents are ingested, THE OllinAI SHALL recompute affected DORA_Metrics incrementally and reflect updated values within 60 seconds of ingestion
7. WHEN insufficient data exists to compute a metric (fewer than 3 data points), THE OllinAI SHALL indicate the metric as "insufficient data" rather than displaying a potentially misleading value
8. IF an Incident included in MTTR computation has no resolution timestamp, THEN THE OllinAI SHALL exclude that Incident from the MTTR average and indicate the count of unresolved Incidents separately

### Requirement 4: Deployment Risk Scoring

**User Story:** As a team lead, I want each deployment scored for risk before and after it occurs, so that I can make informed decisions about when and how to deploy.

#### Acceptance Criteria

1. WHEN a Deployment_Event is ingested, THE OllinAI SHALL compute a Risk_Score (low, medium, high, critical) within 5 seconds of ingestion
2. THE OllinAI SHALL compute Risk_Score based on the following Risk_Factors using a 90-day lookback window: historical Change_Failure_Rate for the affected Service, change size (total files modified and total lines modified), deployment time-of-day and day-of-week historical failure correlation for the affected Service, and author historical failure rate
3. THE OllinAI SHALL weight each Risk_Factor using values between 0 and 1 inclusive that must sum to 1.0, with the following defaults: Change_Failure_Rate 0.35, change size 0.25, deployment timing 0.20, and author failure rate 0.20
4. WHERE a Tenant configures custom Risk_Factor weights, THE OllinAI SHALL validate that all weights are between 0 and 1 inclusive and sum to 1.0, and apply the custom weights for all Risk_Score computations for that Tenant
5. WHEN a Deployment_Event has fewer than 10 historical deployments for the affected Service, THE OllinAI SHALL use organization-wide baseline statistics as fallback data for Risk_Score computation
6. THE OllinAI SHALL expose a pre-deployment risk assessment API that accepts proposed change metadata (affected Service, author identifier, change size in files and lines, and planned deployment timestamp) and returns a projected Risk_Score within 5 seconds without persisting a Deployment_Event
7. IF Risk_Score computation fails due to unavailable data or internal error, THEN THE OllinAI SHALL mark the Risk_Score as "indeterminate", persist the Deployment_Event, and notify the Tenant administrator
8. IF a Tenant submits custom Risk_Factor weights that do not sum to 1.0 or contain values outside the range 0 to 1, THEN THE OllinAI SHALL reject the configuration with an error message indicating the validation failure

### Requirement 5: Actionable Recommendations

**User Story:** As an engineering manager, I want specific recommendations to reduce deployment risk, so that my teams can improve their change failure rate over time.

#### Acceptance Criteria

1. WHEN a Deployment_Event receives a Risk_Score of high or critical, THE OllinAI SHALL generate at least one Recommendation within 10 seconds, targeting the Risk_Factor with the highest weighted contribution to the computed Risk_Score
2. THE OllinAI SHALL generate Recommendations in the following categories: reduce change size (triggered when change size Risk_Factor is dominant), adjust deployment timing (triggered when time-of-day or day-of-week Risk_Factor is dominant), increase review coverage (triggered when author historical failure rate Risk_Factor is dominant), split into smaller services (triggered when a Service exceeds 5 correlated Incidents in a 30-day window), and add canary deployment (triggered when a Service has no post-deploy observation configured and Change_Failure_Rate exceeds 15 percent)
3. WHEN a Team's Change_Failure_Rate increases by more than 20 percentage points over a 7-day rolling window and the Team has at least 5 Deployment_Events within that window, THE OllinAI SHALL generate a trend-based Recommendation visible on the Team's dashboard within 60 seconds of the triggering Deployment_Event correlation
4. THE OllinAI SHALL present each Recommendation with a supporting data summary including the specific metric values that triggered the Recommendation, the time range evaluated, and the affected Service or Team
5. WHEN a user dismisses a Recommendation, THE OllinAI SHALL record the dismissal and suppress the same Recommendation category for the same Team and Service combination for 14 days
6. IF the OllinAI cannot generate a Recommendation for a high or critical Risk_Score Deployment_Event because no category mapping applies to the dominant Risk_Factor, THEN THE OllinAI SHALL log the event for review and display a generic risk alert referencing the unmapped Risk_Factor

### Requirement 6: Team and Service Management

**User Story:** As an engineering manager, I want to organize services by team ownership, so that metrics and risk scores are attributed to the correct teams.

#### Acceptance Criteria

1. THE OllinAI SHALL allow Tenant administrators to create, update, and archive Teams with a unique name (1 to 100 characters) and a member list of up to 200 members per Team
2. THE OllinAI SHALL allow Tenant administrators to create and assign Services to exactly one owning Team, with a Service name of 1 to 150 characters unique within the Tenant
3. WHEN a Service ownership changes, THE OllinAI SHALL attribute future Deployment_Events to the new owning Team while retaining historical attribution
4. THE OllinAI SHALL support a minimum of 200 Services and 50 Teams per Tenant
5. IF a Deployment_Event references a Service not yet registered, THEN THE OllinAI SHALL auto-create the Service as unassigned and notify Tenant administrators via an in-application notification within 60 seconds of ingestion
6. IF a Tenant administrator attempts to archive a Team that still owns one or more Services, THEN THE OllinAI SHALL reject the archive operation and return an error message indicating the Services that must be reassigned before archiving

### Requirement 7: Multi-Tenant Data Isolation and Access Control

**User Story:** As a security-conscious buyer, I want strict data isolation between organizations, so that my deployment and incident data is never visible to other tenants.

#### Acceptance Criteria

1. THE OllinAI SHALL isolate all data at the Tenant level such that no API request or query can access data belonging to a different Tenant
2. THE OllinAI SHALL enforce role-based access control with the following roles and permissions: Tenant Admin (full read and write access to all Tenant resources including Team, Service, integration, and subscription management), Team Lead (read and write access to Deployment_Events, Incidents, and configurations for their assigned Teams and Services), and Viewer (read-only access to dashboards, metrics, and reports with no ability to create, modify, or delete any resource)
3. WHEN a user with Viewer role attempts a create, update, or delete operation on any resource, THE OllinAI SHALL reject the request with an HTTP 403 response and an error message indicating insufficient permissions
4. THE OllinAI SHALL authenticate all API requests using JWT tokens with a maximum validity of 1 hour
5. WHEN a JWT token is missing, expired, or malformed, THE OllinAI SHALL reject the request with an HTTP 401 response
6. THE OllinAI SHALL log all authentication failures and access denied events including actor identity (or source IP if unauthenticated), attempted action, target resource, timestamp, and outcome for security audit purposes
7. WHEN a user with Team Lead role attempts to modify a resource belonging to a Team they are not assigned to, THE OllinAI SHALL reject the request with an HTTP 403 response

### Requirement 8: Subscription Tier Enforcement

**User Story:** As a product owner, I want usage limits enforced by subscription tier, so that the business model is sustainable and upgrades are incentivized.

#### Acceptance Criteria

1. THE OllinAI SHALL enforce the following Starter tier limits: maximum 5 Services, 30-day Retention_Period, and DORA_Metrics only (no Risk_Score or Recommendations)
2. THE OllinAI SHALL enforce the following Pro tier limits: no tier-imposed Service limit, 90-day Retention_Period, Risk_Score, incident correlation, and Recommendations
3. THE OllinAI SHALL enforce the following Enterprise tier limits: no tier-imposed Service limit, no tier-imposed Retention_Period limit, SSO integration, audit logs, custom integrations, API access, Data_Residency_Mode, and Build_Attestation access
4. WHEN a Tenant on the Starter tier attempts to register a Service that would exceed the 5-Service maximum, THE OllinAI SHALL reject the request with a message indicating the tier limit and suggesting an upgrade
5. IF a Tenant on the Starter tier requests a feature restricted to a higher tier (Risk_Score, Recommendations, or incident correlation), THEN THE OllinAI SHALL reject the request with a message indicating the feature is unavailable on the current tier and suggesting an upgrade
6. THE OllinAI SHALL archive data older than the Retention_Period for the Tenant's active subscription tier by running the archival process at least once every 24 hours
7. WHEN a Tenant upgrades their subscription tier, THE OllinAI SHALL apply the new tier limits within 60 seconds without requiring data re-ingestion
8. WHEN a Tenant downgrades their subscription tier, THE OllinAI SHALL apply the new tier limits within 60 seconds, restrict access to features not included in the new tier, and retain existing Services and data without deletion until the new Retention_Period policy archives them

### Requirement 9: Dashboard and Visualization

**User Story:** As an engineering manager, I want an intuitive dashboard showing deployment trends, risk distribution, and DORA metrics, so that I can quickly assess team health.

#### Acceptance Criteria

1. THE OllinAI SHALL display a summary dashboard showing all four DORA_Metrics with trend indicators for the selected time range, where a trend is computed by comparing the current period to the immediately preceding period of equal length: "improving" if the metric improved by more than 10 percent, "degrading" if the metric worsened by more than 10 percent, and "stable" otherwise
2. WHEN a user selects a specific Service or Team, THE OllinAI SHALL filter all dashboard visualizations to display data for the selected scope only
3. THE OllinAI SHALL display a deployment timeline view showing Deployment_Events with color-coded Risk_Scores (one distinct color per severity level: low, medium, high, critical) and correlated Incidents overlaid
4. THE OllinAI SHALL display Risk_Score distribution as a histogram grouped by severity level for the selected time range
5. WHEN new Deployment_Events are ingested, THE OllinAI SHALL update the dashboard within 30 seconds without requiring a manual page refresh
6. THE OllinAI SHALL render the dashboard within 3 seconds for Tenants with up to 10,000 Deployment_Events in the selected time range
7. THE OllinAI SHALL provide a time range selector with a default of 30 days and selectable options of 7, 14, 30, 60, and 90 days, constrained by the Tenant's Retention_Period
8. IF fewer than 3 Deployment_Events exist for the selected scope and time range, THEN THE OllinAI SHALL display an "insufficient data" indicator in place of trend indicators and histograms rather than rendering potentially misleading visualizations
9. WHEN no Service or Team filter is selected, THE OllinAI SHALL display aggregated data across all Services and Teams within the Tenant
10. THE OllinAI SHALL display a CI/CD Security Posture section on the dashboard that maps the Tenant's detected risks, agent findings, and rule matches to the OWASP Top 10 CI/CD Security Risks framework (CICD-SEC-1 through CICD-SEC-10), showing for each risk category: a coverage status (monitored, partially monitored, or not monitored), the count of related detections in the selected time range, and the most recent detection timestamp
11. THE OllinAI SHALL compute the CI/CD Security Posture coverage status per OWASP risk category as follows: CICD-SEC-3 (Dependency Chain Abuse) maps to supply chain credential exfiltration detections and unknown network connection flags; CICD-SEC-6 (Insufficient Credential Hygiene) maps to credential file access detections from the eBPF agent; CICD-SEC-9 (Improper Artifact Integrity Validation) maps to Build_Attestation generation status; CICD-SEC-10 (Insufficient Logging and Visibility) maps to OllinAI_Agent deployment coverage across the Tenant's pipelines; remaining categories (CICD-SEC-1, 2, 4, 5, 7, 8) SHALL display as "not monitored" with a description of what monitoring would entail and a link to relevant documentation
12. WHEN a user clicks on an OWASP risk category in the CI/CD Security Posture section, THE OllinAI SHALL navigate to a detail view showing all related detections, affected pipelines, and recommended mitigations for that category

### Requirement 10: Integration Management

**User Story:** As a platform engineer, I want to configure integrations with my existing CI/CD and incident management tools, so that data flows into OllinAI automatically.

#### Acceptance Criteria

1. THE OllinAI SHALL provide open-source pipeline actions/steps for GitHub Actions, GitLab CI, Jenkins, and CircleCI that post Deployment_Events to the OllinAI Webhook endpoint
2. THE OllinAI SHALL provide pre-built Webhook receivers for PagerDuty and OpsGenie incidents
3. THE OllinAI SHALL provide a generic Webhook endpoint accepting a documented JSON schema for custom integration sources including unsupported CI/CD systems, with a maximum accepted payload size of 1 MB per request
4. WHEN an integration is configured, THE OllinAI SHALL validate connectivity by sending a test event and returning a success or failure response within 10 seconds indicating whether the event was received and processed
5. THE OllinAI SHALL authenticate incoming Webhooks using HMAC-SHA256 signature verification with a per-integration secret key of at least 32 bytes
6. IF a Webhook signature verification fails, THEN THE OllinAI SHALL reject the payload with an HTTP 401 response and log the event for security audit
7. THE OllinAI SHALL allow Tenant administrators to generate, rotate, and revoke integration secret keys from the management interface
8. IF an authenticated Webhook payload passes signature verification but fails processing due to malformed content or an internal error, THEN THE OllinAI SHALL return an HTTP 422 response with an error message indicating the reason for the processing failure and SHALL NOT acknowledge the event as successfully ingested

### Requirement 11: Data Export and API Access

**User Story:** As a VP of Engineering, I want to export deployment and metrics data via API, so that I can incorporate it into executive reporting and other internal tools.

#### Acceptance Criteria

1. WHERE a Tenant has Enterprise tier access, THE OllinAI SHALL provide a RESTful API returning Deployment_Events, Incidents, DORA_Metrics, and Risk_Scores in JSON format within 5 seconds per request for result sets up to 100 records
2. THE OllinAI SHALL support pagination for API responses with a default page size of 25 records and a maximum page size of 100 records, and SHALL include in each response the total record count, current page number, and an indicator of whether additional pages exist
3. THE OllinAI SHALL rate-limit API requests to 100 requests per minute per Tenant to protect system stability
4. WHEN an API request exceeds the rate limit, THE OllinAI SHALL return an HTTP 429 response with a Retry-After header indicating when the next request can be made
5. THE OllinAI SHALL support filtering API results by Service, Team, time range (bounded by the Tenant's Retention_Period), and Risk_Score severity
6. IF an API request contains invalid filter parameters (non-existent Service, non-existent Team, malformed time range, or invalid Risk_Score severity value), THEN THE OllinAI SHALL return an HTTP 400 response with an error message identifying the invalid parameters
7. IF an API request with valid filters returns no matching records, THEN THE OllinAI SHALL return an HTTP 200 response with an empty data array and a total record count of zero

### Requirement 12: Audit Logging

**User Story:** As a compliance officer, I want a complete audit trail of configuration changes and data access, so that I can demonstrate regulatory compliance.

#### Acceptance Criteria

1. WHERE a Tenant has Enterprise tier access, THE OllinAI SHALL log all configuration changes including Team, Service, integration, and subscription modifications, as well as data access events including API data exports, audit log retrievals, and bulk data queries
2. THE OllinAI SHALL record the following for each audit event: actor identity, action performed, target resource, timestamp in UTC with millisecond precision, source IP address, and outcome (success or failure)
3. THE OllinAI SHALL retain audit logs for a minimum of 365 days regardless of the Tenant's data Retention_Period
4. WHEN a Tenant Admin requests audit logs, THE OllinAI SHALL return paginated results (maximum 100 records per page) filterable by actor, action type, target resource, and time range, within 5 seconds for queries spanning up to 90 days
5. THE OllinAI SHALL ensure audit logs are append-only and cannot be modified or deleted by any user including Tenant administrators
6. IF a Tenant downgrades from Enterprise tier, THEN THE OllinAI SHALL retain existing audit logs for the full 365-day retention period but cease recording new audit events until Enterprise tier access is restored

### Requirement 13: eBPF Agent — Deployment Runtime Profiling (Phase 2)

**User Story:** As a platform engineer, I want deep runtime visibility into what happens during my CI/CD pipeline execution, so that I can detect anomalies, supply chain risks, and resource issues during deployment.

#### Acceptance Criteria

1. THE OllinAI_Agent SHALL be distributable as both a standalone binary for VM-based runners and a container sidecar for Kubernetes-based runners
2. WHEN the OllinAI_Agent starts, THE OllinAI_Agent SHALL perform a pre-flight kernel compatibility check verifying: Linux kernel version is 5.15 or later on amd64 (6.1 or later on arm64), BPF syscall is available, required BPF helper functions are supported, and the process has CAP_BPF and CAP_PERFMON capabilities (or CAP_SYS_ADMIN on kernels prior to 5.8), and SHALL report the check result (pass, degraded, or unsupported) to the Collector_API before attaching any probes
3. IF the pre-flight kernel compatibility check returns "unsupported" (kernel version too old or BPF syscall unavailable), THEN THE OllinAI_Agent SHALL log a clear error message identifying the minimum kernel version required for the detected architecture, operate exclusively in userspace fallback mode, and include the kernel version and architecture in all telemetry metadata
4. IF the pre-flight kernel compatibility check returns "degraded" (kernel version supported but capabilities are insufficient), THEN THE OllinAI_Agent SHALL log a warning identifying the missing capabilities, operate in userspace fallback mode, and recommend the specific capability grants needed for full eBPF operation
5. WHEN the OllinAI_Agent is active during a pipeline execution, THE OllinAI_Agent SHALL capture process tree activity, network connections, file system writes, and resource consumption (CPU and memory utilization) using eBPF probes
3. WHEN the OllinAI_Agent detects outbound network connections to domains not present in the baseline for that pipeline, THE OllinAI_Agent SHALL flag the connection as a potential supply chain anomaly and include the destination domain, port, and initiating process in the anomaly report
4. WHEN the OllinAI_Agent is active during a pipeline execution, THE OllinAI_Agent SHALL track the full Process_Ancestry tree for every observed process, recording parent-child relationships, command arguments, and working directories
5. WHEN the OllinAI_Agent detects credential file access (reading of files matching patterns such as ~/.aws/credentials, ~/.docker/config.json, GITHUB_TOKEN environment variable reads, or SSH private keys) from a process whose Process_Ancestry includes a package installation command (npm install, pip install, go get, cargo build, or equivalent), THE OllinAI_Agent SHALL flag the event as a high-confidence supply chain credential exfiltration attempt
6. WHEN a pipeline execution completes, THE OllinAI_Agent SHALL generate a Build_Attestation document containing: all processes executed with Process_Ancestry, all outbound network connections with destination and initiating process, all file writes to sensitive paths, and a SHA-256 digest of the complete telemetry stream
7. THE OllinAI_Agent SHALL cryptographically sign the Build_Attestation using a per-agent signing key and transmit the attestation to the Collector_API alongside the telemetry batch
8. WHEN a pipeline execution consumes more than 2x the rolling average CPU or memory compared to the previous 10 executions of the same pipeline, THE OllinAI_Agent SHALL flag a resource anomaly
9. THE OllinAI_Agent SHALL transmit collected telemetry to the OllinAI Collector_API in batches of no more than 500 events each, with a maximum latency of 10 seconds from event capture to batch transmission
10. THE OllinAI_Agent SHALL add no more than 5 percent overhead to pipeline execution time when the pipeline completes within 2x its rolling average duration and generates fewer than 10,000 telemetry events per minute
11. THE OllinAI SHALL incorporate eBPF runtime profiling data (supply chain anomaly flags and resource anomaly flags) as Risk_Factors in the Risk_Score computation for the associated Deployment_Event
12. IF the OllinAI_Agent fails to attach eBPF probes (due to insufficient kernel version or permissions), THEN THE OllinAI_Agent SHALL fall back to userspace collection of process tree activity and network connections, log a degraded-mode warning, and report the degraded status to the Collector_API
13. WHEN a pipeline has fewer than 5 completed executions with the OllinAI_Agent active, THE OllinAI_Agent SHALL operate in baseline-building mode, collecting telemetry without generating anomaly flags, and SHALL indicate that anomaly detection is not yet active for that pipeline
14. IF the OllinAI_Agent cannot reach the Collector_API, THEN THE OllinAI_Agent SHALL buffer telemetry locally for up to 5 minutes and retry transmission every 30 seconds until connectivity is restored or the buffer is full
15. IF the OllinAI_Agent local buffer reaches capacity while the Collector_API is unreachable, THEN THE OllinAI_Agent SHALL drop the oldest buffered events first and record the count of dropped events for inclusion in the next successful transmission

### Requirement 14: eBPF Agent — Post-Deploy Canary Observation (Phase 2)

**User Story:** As an SRE, I want automated observation of my services in the first minutes after deployment, so that I can detect regressions at the kernel level before they become customer-facing incidents.

#### Acceptance Criteria

1. WHEN a new deployment is detected on a monitored host, THE OllinAI_Agent SHALL observe the deployed Service's syscall profile, network connections, and error signals for a configurable observation window (default 5 minutes, configurable from 1 minute to 60 minutes)
2. WHEN the observation window completes, THE OllinAI_Agent SHALL compare the post-deployment syscall profile against the rolling baseline (computed from the previous 10 successful deployments for that Service) and flag deviations exceeding a configurable threshold (default 30 percent divergence, configurable from 5 to 95 percent)
3. WHEN the OllinAI_Agent detects kernel-level error signals (OOM kills, segmentation faults, or 5 or more failed connection attempts within any 30-second interval) within the observation window, THE OllinAI_Agent SHALL report an early warning event to OllinAI within 5 seconds of detection
4. WHEN an early warning event is received from the OllinAI_Agent, THE OllinAI SHALL correlate it with the triggering Deployment_Event and escalate the Risk_Score to critical
5. WHEN no anomalies are detected during the observation window, THE OllinAI_Agent SHALL report a healthy canary result that reduces the Service's Risk_Score by one severity level for the next Risk_Score computation (minimum: low)
6. THE OllinAI_Agent SHALL support deployment detection via Kubernetes rollout events, systemd service restarts, and process binary change detection
7. WHERE a Tenant has Enterprise tier access, THE OllinAI SHALL provide access to the full eBPF Intelligence Layer including runtime profiling and post-deploy canary observation
8. IF the OllinAI_Agent cannot deliver an early warning or healthy canary report to OllinAI after 3 retry attempts, THEN THE OllinAI_Agent SHALL queue the report locally and retry delivery every 60 seconds until successful or until 24 hours have elapsed
9. IF a Service has fewer than 10 historical deployments and no rolling baseline is available, THEN THE OllinAI_Agent SHALL observe and report raw telemetry to OllinAI without performing deviation comparison and mark the result as baseline-building

### Requirement 15: AIOps ML Engine — Continuous Training and Inference Infrastructure

**User Story:** As a platform engineer, I want OllinAI to continuously train ML models on our deployment and incident history, so that risk assessments and operational decisions are driven by learned patterns rather than static rules alone.

#### Acceptance Criteria

1. THE AIOps_Engine SHALL maintain a Training_Pipeline that retrains ML_Models on newly ingested Deployment_Events, Incidents, and telemetry data at a configurable interval (default every 24 hours, configurable from 6 hours to 7 days)
2. WHEN the Training_Pipeline completes a training cycle, THE AIOps_Engine SHALL validate the new ML_Model against a holdout dataset comprising 20 percent of historical data and promote the model to production only if its accuracy exceeds the currently deployed model by at least 1 percentage point or maintains equivalent accuracy
3. THE AIOps_Engine SHALL store all trained ML_Models in the Model_Registry with the following metadata: model version identifier, training timestamp, dataset version, feature set used, and evaluation metrics (precision, recall, F1 score, and AUC-ROC)
4. WHEN the AIOps_Engine serves a prediction, THE AIOps_Engine SHALL return a Prediction_Score between 0.0 and 1.0 with inference latency not exceeding 200 milliseconds per request
5. THE AIOps_Engine SHALL compute a Drift_Score at each training interval by comparing the statistical distribution of incoming Feature_Vectors against the training dataset distribution of the active ML_Model
6. WHEN the Drift_Score exceeds 0.7, THE AIOps_Engine SHALL trigger an immediate retraining cycle outside the scheduled interval and notify Tenant administrators that model drift was detected
7. IF the Training_Pipeline fails to produce a model that meets the promotion threshold after 3 consecutive training cycles, THEN THE AIOps_Engine SHALL alert Tenant administrators, continue serving the current production model, and log the failure reason for review
8. THE AIOps_Engine SHALL require a minimum of 100 Deployment_Events and 10 correlated Incidents for a Tenant before activating ML-based predictions, serving rule-based Risk_Scores until sufficient data is available
9. WHERE a Tenant has Pro or Enterprise tier access, THE OllinAI SHALL provide access to AIOps_Engine predictions and model performance metrics
10. THE AIOps_Engine SHALL construct Feature_Vectors from the following inputs: change size metrics, deployment timing patterns, Service historical failure rates, author historical failure rates, inter-service dependency graph, time since last incident for the affected Service, and eBPF telemetry signals (when available)

### Requirement 16: Predictive Intelligence — Incident Prediction, Anomaly Detection, and Root Cause Analysis

**User Story:** As an SRE, I want ML-driven predictions that alert me to likely incidents before they occur and identify the most probable root cause when they do, so that I can prevent outages and reduce mean time to resolution.

#### Acceptance Criteria

1. WHEN a Deployment_Event is ingested, THE AIOps_Engine SHALL generate a Prediction_Score representing the probability that the deployment will result in a correlated Incident within the next 60 minutes, and surface a pre-incident alert when the Prediction_Score exceeds a configurable threshold (default 0.75, configurable from 0.5 to 0.95)
2. WHILE the AIOps_Engine is active, THE AIOps_Engine SHALL perform continuous anomaly detection on Service metrics (error rates, latency percentiles, throughput) and flag Anomaly_Signals when observed values deviate from the ML_Model's learned normal behavior by more than 3 standard deviations of the model's prediction interval
3. WHEN an Anomaly_Signal is detected for a Service, THE AIOps_Engine SHALL correlate it with Deployment_Events within the preceding Correlation_Window and generate an early warning notification within 30 seconds of detection, including the anomalous metric name, current value, expected range, and correlated Deployment_Event identifiers
4. WHEN an Incident is confirmed (persisted with a correlated Deployment_Event), THE AIOps_Engine SHALL perform ML-based root cause analysis by ranking all Deployment_Events within the Correlation_Window using learned causal patterns — including change content similarity to past incident-causing deployments, Service dependency propagation paths, and temporal proximity — and present the top 3 most probable root causes with confidence scores
5. THE AIOps_Engine SHALL achieve a minimum precision of 70 percent and recall of 60 percent for incident prediction (Prediction_Score exceeding threshold) measured on a rolling 30-day evaluation window, and report these metrics to Tenant administrators monthly
6. IF the AIOps_Engine's incident prediction precision falls below 60 percent on the rolling evaluation window, THEN THE AIOps_Engine SHALL automatically suppress pre-incident alerts, trigger an immediate retraining cycle, and notify Tenant administrators that predictive alerting has been temporarily disabled
7. WHEN the AIOps_Engine generates a root cause ranking, THE AIOps_Engine SHALL include an explanation summary for each ranked Deployment_Event describing the contributing factors (e.g., "similar change pattern caused incident X on date Y", "deployed to shared dependency of affected Service")
8. THE AIOps_Engine SHALL distinguish ML-based Prediction_Scores from rule-based Risk_Scores by labeling each assessment with its source (ml_model or rule_engine) in all API responses and dashboard displays

### Requirement 17: Automated Decision-Making — ML-Driven Remediation and Deployment Gating

**User Story:** As a team lead, I want the platform to recommend or automatically execute remediation actions based on ML predictions, so that my team can respond faster to deployment risks and reduce manual intervention during incidents.

#### Acceptance Criteria

1. WHEN a Deployment_Event receives a Prediction_Score above the configured alert threshold and a Risk_Score of critical, THE AIOps_Engine SHALL generate a Remediation_Action recommendation within 10 seconds, selecting from the following action types: rollback deployment, halt canary promotion, scale up affected Service, or notify on-call engineer
2. THE AIOps_Engine SHALL select the Remediation_Action type based on ML_Model inference over historical remediation outcomes — learning which action type resolved similar past incidents most effectively for the affected Service and deployment pattern
3. WHERE a Tenant configures automated remediation mode for a Service, THE AIOps_Engine SHALL execute the recommended Remediation_Action automatically (via integration with the CI/CD pipeline) when the Prediction_Score exceeds 0.9 and the ML_Model's confidence in the selected action exceeds 0.85
4. WHEN the AIOps_Engine executes an automated Remediation_Action, THE AIOps_Engine SHALL log the action with full context (triggering Deployment_Event, Prediction_Score, selected action, ML_Model version, and confidence score) and notify the Service's owning Team within 5 seconds of execution
5. IF an automated Remediation_Action fails to execute (CI/CD integration error, timeout, or permission failure), THEN THE AIOps_Engine SHALL escalate to a manual recommendation, notify the on-call engineer, and record the failure reason
6. THE AIOps_Engine SHALL provide a deployment gate API that accepts a proposed deployment and returns a gate decision (proceed, warn, or block) based on the ML_Model's Prediction_Score combined with the rule-based Risk_Score, using configurable thresholds per Service (defaults: proceed below 0.5, warn between 0.5 and 0.8, block above 0.8)
7. WHEN the deployment gate returns a "block" decision, THE AIOps_Engine SHALL include in the response the specific Risk_Factors and ML-predicted outcomes that contributed to the decision, along with suggested mitigations to reduce the Prediction_Score below the block threshold
8. WHERE a Tenant has not enabled automated remediation mode, THE AIOps_Engine SHALL operate in recommendation-only mode, presenting Remediation_Actions as suggestions without executing them
9. THE AIOps_Engine SHALL track the outcome of all Remediation_Actions (whether executed automatically or manually by operators) and incorporate the outcomes as training signals for subsequent ML_Model retraining cycles
10. WHEN the AIOps_Engine recommends a Remediation_Action that contradicts the last operator decision for the same Service and action type, THE AIOps_Engine SHALL flag the recommendation with a conflict indicator and require explicit operator confirmation before automated execution regardless of confidence scores

### Requirement 18: Externalized Rule Engine — OCI-Based Detection Rules

**User Story:** As a security engineer, I want detection rules to be updatable independently of the agent binary, so that new threat patterns can be deployed across my fleet without redeploying agents.

#### Acceptance Criteria

1. THE OllinAI_Agent SHALL load detection rules from a Rule_Bundle distributed as an OCI artifact, separate from the agent binary
2. THE OllinAI_Agent SHALL check for Rule_Bundle updates at a configurable interval (default every 6 hours, configurable from 1 hour to 24 hours) and apply new rules without requiring an agent restart
3. THE OllinAI SHALL provide a set of baseline detection rules covering: credential access from package installation descendants, outbound data exfiltration patterns, cryptocurrency miner process signatures, and known malicious domain connections
4. THE OllinAI SHALL allow Tenant administrators to author custom detection rules using a declarative YAML-based rule language that supports matching on Process_Ancestry patterns, file access paths, network destinations, and resource consumption thresholds
5. WHEN a detection rule matches during a pipeline execution, THE OllinAI_Agent SHALL emit an alert event containing the rule identifier, matched process details, Process_Ancestry chain, and severity level (info, warning, critical)
6. THE OllinAI_Agent SHALL support a monitor-only mode where detection rule matches are logged and reported but do not terminate the pipeline, and an enforcement mode where critical-severity matches terminate the job immediately
7. IF the OllinAI_Agent cannot fetch a Rule_Bundle update (network failure or invalid artifact), THEN THE OllinAI_Agent SHALL continue operating with the last successfully loaded Rule_Bundle and log the update failure
8. THE OllinAI SHALL version Rule_Bundles using semantic versioning and maintain the 3 most recent versions in the registry to allow rollback if a new rule causes false positives

### Requirement 19: Data Residency and Telemetry Routing

**User Story:** As a security-conscious Enterprise buyer, I want to ensure that sensitive CI/CD telemetry data never leaves my cloud environment, so that I can meet data residency and compliance requirements.

#### Acceptance Criteria

1. WHERE a Tenant has Enterprise tier access and enables Data_Residency_Mode, THE OllinAI_Agent SHALL transmit telemetry and Build_Attestations directly to the Tenant's designated Amazon S3 bucket rather than to the OllinAI Collector_API
2. WHEN Data_Residency_Mode is enabled, THE OllinAI SHALL read telemetry from the Tenant's S3 bucket for processing (risk scoring, anomaly detection, ML inference) using an AWS cross-account IAM role assumed via STS, with the Tenant providing the role ARN during configuration
3. THE OllinAI SHALL NOT persist raw telemetry data on OllinAI-managed infrastructure when Data_Residency_Mode is enabled, processing data in-transit only and storing only derived metrics (Risk_Scores, Anomaly_Signals, Prediction_Scores)
4. WHEN a Tenant configures Data_Residency_Mode, THE OllinAI SHALL validate S3 bucket connectivity and IAM role assume permissions by writing and reading a test object, and confirm successful configuration within 30 seconds
5. IF the OllinAI_Agent cannot write to the Tenant's designated S3 bucket, THEN THE OllinAI_Agent SHALL buffer telemetry locally (up to 5 minutes) and retry, following the same buffer overflow behavior defined in Requirement 13
6. THE OllinAI SHALL support Data_Residency_Mode for Amazon S3 buckets in any AWS region, and SHALL use the same region as the Tenant's bucket for processing Lambda functions to minimize data transfer latency and costs
